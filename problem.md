2. 语义理解的“降维” (Semantic Capability Gap)

    原因：text2vec-base-chinese 是一个轻量级模型（大概只有 1 亿多参数）。而你原本想用的 Google text-embedding-004 或 DeepSeek 的向量模型，参数量是它的几十倍甚至上百倍。

    ## 表现：

    对于简单的字面意思（如“比赛什么时间开始”），本地模型完全能胜任。

    但对于复杂的跨句推理或高度抽象的语义（比如用户问：“有没有对大一新生比较友好，且不需要编程基础的赛道？”），本地模型生成的向量很难精确捕捉到这种长程的逻辑关联，导致检索出来的结果不够贴切。

3. 多语言与专业术语的短板

    原因：这是一个专门针对“通用中文”微调的模型。

    后果：如果你的竞赛 PDF 里包含大量英文缩写、前沿的计算机专业词汇或特定的代码框架名称，这个模型在进行 Tokenize（分词）时，可能会把它们切碎，导致对这些专业术语的特征提取不够敏锐。

4. CPU 算力瓶颈 (Scalability limits)

    现状：你现在处理 200 多个文本块，CPU 几秒钟或几十秒就跑完了。

    未来风险：如果高教授让你把全校历年 10,000 份竞赛通知全加进去（变成几万个 Chunks），在没有 GPU（显卡）加速的情况下，纯靠 CPU 跑本地 Embedding 可能会耗费几个小时，系统扩展性受到物理硬件的制约。
